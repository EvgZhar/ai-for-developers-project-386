## Проектирование приложения
Проект выполняется в подходе Design First: сначала фиксируем поведение системы и API-контракт, и только потом переходим к реализации. Фронтенд и бэкенд реализуются раздельно и взаимодействуют через API-контракт. Контракт в этом шаге задает границу между частями приложения и остается единым источником правды для реализации.

Спецификация API - это единый источник правды для обеих частей приложения. По ней можно независимо реализовывать фронтенд и бэкенд, не тратя ресурсы на анализ внутренней реализации соседней части. Это ускоряет работу агентов, снижает расход токенов и уменьшает требования к модели. При изменениях достаточно обновить контракт и синхронно внести правки в обе части.

Для этого возьмем TypeSpec - удобный инструмент для создания OpenAPI спецификации. Также он позволяет генерировать код практически в любой язык. Он похож на TypeScript, но это лишь визуальное сходство, TypeSpec не полноценный язык программирования.

Начиная с этого шага мы всю работу выполняем с помощью агентов: описываем план, уточняем детали, реализуем. В идеале, мы не должны написать ни одной строчки кода вручную.

## Стартовая страница и управление профилем

Авторизация не предусмотрена. Сессия хранится в `localStorage` (ключ `currentProfileId`).

При первом визите или после «Сменить пользователя» отображается стартовая страница с двумя вариантами:

1. **«Я здесь впервые»** — форма создания гостя (React Hook Form + Zod):
   - `name` — обязательное, проверка уникальности среди существующих guest-профилей
   - `locale` — select из `Intl.supportedValuesOf('locale')`
   - `timeZone` — select из `Intl.supportedValuesOf('timeZone')`
   - После `POST /api/profiles` → авто-выбор нового профиля → редирект в приложение

2. **«Я уже был»** — список всех существующих профилей (`GET /api/profiles`):
   - Admin-профиль — с меткой «Владелец»
   - Guest-профили — по имени (если имена совпадают, показывается дата создания)
   - Клик → сохранение `profileId` в `localStorage` → редирект

В приложении — кнопка «Сменить пользователя»:
- Очищает `localStorage`
- Вызывает `queryClient.clear()` (TanStack Query — полный сброс кеша, чтобы данные старого пользователя не просочились)
- Редирект на стартовую страницу

## Сущности (в соответствии с API-контрактом)

### Profile (профиль пользователя)
Профиль — это лёгкая сущность для разделения ролей. Не требует пароля или полноценной регистрации.

- `id` (string) — уникальный идентификатор
- `name` (string) — отображаемое имя
- `type` (enum: `admin`, `guest`) — роль: admin = владелец календаря, guest = гость
- `locale` (string) — код локали, например `ru-RU`, `en-US`
- `timeZone` (string) — IANA timezone, например `Europe/Moscow`

**Admin** — один предопределённый профиль (создаётся при первом запуске).  
**Guest** — создаётся анонимно при первом входе (без пароля, только имя + locale + timezone).

### Schedule (расписание)
Единое расписание рабочего времени владельца календаря. Хранит шаблон по дням недели.

- `timeZone` (string) — таймзона, в которой заданы интервалы
- `days` (массив DaySchedule) — расписание на каждый день недели

#### DaySchedule
- `dayOfWeek` (int, 0–6) — 0 = воскресенье, 1 = понедельник, …, 6 = суббота
- `intervals` (массив TimeInterval) — непересекающиеся рабочие интервалы

#### TimeInterval
- `startTime` (string, HH:mm) — начало интервала в локальном времени
- `endTime` (string, HH:mm) — конец интервала

#### Валидация расписания (Schedule)

Правила проверяются на бэкенде; на фронтенде — дублируются для мгновенного фидбека:

| Правило | Пояснение |
|---|---|
| `timeZone` — обязательна, валидная IANA | Список из `Intl.supportedValuesOf('timeZone')` |
| `dayOfWeek` — 0..6, все значения уникальны | Два одинаковых дня → ошибка |
| `startTime` < `endTime` | Равные или start > end → ошибка |
| `endTime` – `startTime` ≥ 15 мин | Минимальная длительность интервала |
| Интервалы **не пересекаются** (включая стык) | `09:00–12:00` и `12:00–15:00` → `SCHEDULE_OVERLAP`. Условие: `existing.start < new.end && new.start < existing.end` |
| Пустой `days` / пустой `intervals` | Допустимо — владелец не работает |

На UI: при пересечении интервалов — подсветка конфликта и предложение объединить соседние, например: «Объединить в 09:00–15:00».

### EventType (тип события)
Создаётся владельцем для описания доступных видов встреч.

- `id` (string) — уникальный идентификатор
- `title` (string) — название типа события
- `description` (string) — описание, показывается гостям
- `durationMinutes` (int, min 15) — длительность в минутах

### AvailableSlot (доступный слот)
Вычисляемая модель, возвращаемая API. Не хранится в БД.

- `startDateTime` (utcDateTime) — начало слота в UTC
- `endDateTime` (utcDateTime) — конец слота в UTC

### Booking (бронирование)
Фиксирует факт встречи гостя с владельцем.

- `id` (string) — уникальный идентификатор
- `eventTypeId` (string) — FK на EventType
- `startDateTime` (utcDateTime) — начало встречи в UTC
- `endDateTime` (utcDateTime) — конец встречи в UTC (start + durationMinutes)
- `guestProfileId` (string) — FK на Profile гостя
- `guestName` (string) — отображаемое имя гостя (дублируется из Profile для удобства)
- `agenda` (string, optional) — заметки к встрече
- `status` (enum: `requested`, `confirmed`, `rejected`, `cancelled`) — жизненный цикл брони
- `createdAt` (utcDateTime) — дата создания
- `cancelledAt` (utcDateTime, optional) — дата отмены

### Жизненный цикл бронирования (Booking Status Flow)

```
requested ──→ confirmed ──→ cancelled
       └──→ rejected
```

- **requested** — гость создал бронь на свободный слот. Слот считается занятым. Админ видит в панели и может подтвердить или отклонить.
- **confirmed** — админ подтвердил встречу. Статус по умолчанию при бронировании (если админ не хочет модерировать вручную, бэкенд может сразу переводить в confirmed).
- **rejected** — админ отклонил бронь. Слот снова становится доступным.
- **cancelled** — отмена после подтверждения (по инициативе гостя или админа). Промежуточный статус `pending cancel` не вводится, так как в системе нет канала уведомлений для двухшагового подтверждения отмены.

## Описание приложения

В проекте есть две условные роли: владелец календаря (admin) и гости. В проекте нет классической регистрации и авторизации. Владелец календаря — один заранее заданный admin-профиль, используемый по умолчанию в админской части. Гость создаёт лёгкий профиль без пароля (указывает имя, locale и timezone) и бронирует слоты.

Владелец календаря может:
- Управлять рабочим расписанием (дни недели, часы работы)
- Создавать, редактировать и удалять типы событий (title, description, durationMinutes)
- Просматривать страницу всех бронирований с фильтрацией по дате и статусу
- Подтверждать или отклонять бронирования (requested → confirmed / rejected)
- Отменять подтверждённые бронирования (confirmed → cancelled)

Гость:
- Выбирает тип события из списка доступных (активных)
- Выбирает свободный слот в ближайшие 14 дней с учётом расписания владельца
- Указывает имя, locale и timezone (создаётся гостевой профиль)
- Создаёт бронирование на выбранный слот
- Просматривает список всех своих бронирований
- Отменяет подтверждённое бронирование (confirmed → cancelled)

## Навигация (Frontend)

| Путь | Доступ | Описание |
|---|---|---|
| `/` | Все | Стартовая страница: выбор существующего профиля или создание нового |
| `/profile/create` | Все | Форма создания гостевого профиля |
| `/book` | Гость | Лендинг: список активных типов событий |
| `/book/{eventTypeId}` | Гость | Шаг 1: выбор даты и времени (календарь со слотами) |
| `/book/{eventTypeId}/info` | Гость | Шаг 2: форма гостя (имя, locale, timezone, agenda) |
| `/booking/{bookingId}` | Гость | Подтверждение / детали брони |
| `/bookings` | Гость | Список всех броней гостя |
| `/admin` | Admin | Dashboard: все брони (фильтр по дате/статусу) |
| `/admin/schedule` | Admin | Управление расписанием (дни недели + интервалы) |
| `/admin/event-types` | Admin | CRUD типов событий (таблица) |

Flow гостя: `/book` → выбор типа → выбор слота → форма гостя → создание → результат.  
Гость может в любой момент вернуться в `/bookings` и управлять своими бронями.

### Смена пользователя

- Кнопка «Сменить пользователя» доступна в шапке приложения на всех страницах
- Сбрасывает `localStorage` + `queryClient.clear()`, редирект на `/`
- При переходе на `/` без `currentProfileId` — показывается стартовая страница

### Админ-панель

Админ-панель доступна только если `currentProfile.type === 'admin'`.

## Запуск dev-окружения

```bash
npm run dev          # Prism (мок-API) + Vite (фронтенд) одновременно
npm run dev:prism    # Только Prism мок-сервер на http://localhost:4010
npm run dev:client   # Только Vite фронтенд на http://localhost:5173
```

- Фронтенд доступен на `http://localhost:5173`
- Мок-API доступен на `http://localhost:4010` (Prism по OpenAPI)
- После запуска можно открыть `http://localhost:5173` и начать разработку

## Порядок разработки

Реализация ведётся в две очереди:

1. **Фронтенд** —独立но, через Prism (мок-сервер по OpenAPI)
2. **Бэкенд** — после фронтенда, по тому же API-контракту
